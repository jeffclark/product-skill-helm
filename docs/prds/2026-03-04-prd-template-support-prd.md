# PRD: PRD Template Support

## Overview

Product Skill Helm generates PRDs using its built-in `prd-writer` agent. When PMs use the tool at organizations with standardized PRD templates, the generated output does not conform to their company's structure. This forces PMs to manually reformat every PRD before sharing internally.

This feature adds support for a project-level PRD template file (`prd-template.md`). When present in the project root, the template governs the structure of all PRDs generated by `/pm:prd`. Every section in the template receives content. Gaps the agent cannot fill trigger a targeted question to the PM.

---

## Problem Statement

PMs at companies with defined PRD standards must reformat every Product Skill Helm PRD output before it can be shared internally. This rework undermines the tool's core value — reducing PM document burden — and creates a friction point that may limit adoption at organizations with strong documentation standards.

**Observed behavior:** PM uses `/pm:prd`, receives a well-structured PRD, then spends 10–20 minutes reformatting headings and section order before sharing with stakeholders.

**Root cause:** The prd-writer agent generates PRDs using its own default structure. It has no awareness of organizational template standards.

---

## Goals & Success Criteria

**Primary goal:** A PM can share a Product Skill Helm PRD directly with their team without reformatting.

**Success criteria:**

1. Every top-level heading (`##`) from `prd-template.md` appears in the output PRD in the same sequence.
2. Every section has either agent-generated content or an explicit skip placeholder (`_[Section left blank — complete before sharing]_`).
3. No top-level sections are added by the agent that do not appear in the template.
4. When no `prd-template.md` is present, `/pm:prd` behavior is unchanged.

**Out of scope for success measurement:** Whether the generated *content* perfectly matches the PM's intent (this is a content quality concern, not a template conformance concern).

---

## Functional Requirements

### FR-001: Template Detection

At the start of `/pm:prd`, check for `prd-template.md` in the project root.

- **File found:** Print exactly: `Template found: prd-template.md — generating PRD using your template structure.` Proceed in template mode.
- **File not found:** Proceed with current default behavior. No message, no error.

### FR-002: Template-Driven Output Structure

In template mode:

1. Parse every top-level heading (`##`) from `prd-template.md` as a required output section.
2. Generate content for each section based on the brainstorm document, product-context.yaml, and session context.
3. Preserve the heading order from the template exactly — do not reorder sections.
4. Preserve subheadings (`###`) within template sections. Generate content under each subheading.
5. Ignore any placeholder text under template headings (treat it as instructional scaffolding, not as content to preserve).

### FR-003: Gap-Filling Dialogue

When the agent cannot generate high-quality content for a section (no relevant information available from context), it must surface this to the PM before writing the file.

**Agent output format for an unfillable section:**

> "I don't have enough context to fill the **[Section Name]** section. Can you provide guidance, or would you like to skip it for now?"

PM response options:
- Any substantive text → agent uses it as content for the section.
- "skip", "skip for now", "leave it blank", or equivalent → section is written with the placeholder: `_[Section left blank — complete before sharing]_`

The agent must ask about *all* unfillable sections before writing the final file (batch the questions, do not interleave with writing).

### FR-004: Graceful Degradation

If `prd-template.md` is absent:
- `/pm:prd` runs exactly as it does today.
- No error, no warning, no mention of templates.
- Existing PRD output format is fully preserved.

### FR-005: Template File Format

`prd-template.md` is a standard markdown file committed to the project root alongside `product-context.yaml`.

- Top-level headings (`##`) define required output sections.
- Subheadings (`###`) define subsections within a section.
- Body text under headings is treated as placeholder/instruction text and is not preserved in output.
- No custom schema, YAML frontmatter, or special syntax required.

---

## Non-Functional Requirements

### NFR-001: Operational Definition of "Match"

"Output matches template" is defined as all three of the following being true:

- (a) Every `##` heading from the template appears in the output in the same order.
- (b) Every section contains either agent-generated content or the skip placeholder.
- (c) No `##` sections appear in the output that are not in the template.

This definition is the acceptance criterion for QA and for the PM's assessment of feature success.

### NFR-002: No Regression for Templateless Projects

Projects without `prd-template.md` must produce identical output to the current behavior. No performance degradation, no changed prompts, no altered defaults.

## Prototype Spike (Required Before Implementation)

**Story 0 — Template Behavior Prototype**

Before implementing FR-002 and FR-003, run the prd-writer agent against 2–3 real-world template structures and document its behavior on ambiguous inputs:

- Template sections with no analogous concept in the brainstorm
- Template headings that use company-specific terminology the agent hasn't seen
- Templates with deep nesting (### and ####)

**Deliverable:** A findings document that answers:
1. Does the agent reliably place content under the correct heading?
2. Under what conditions does it fail to recognize that it lacks context (i.e., when does it generate low-quality filler instead of asking)?
3. Are there heading naming patterns that cause the agent to conflate sections?

Findings from the spike inform the implementation approach for FR-002 and FR-003. The spike **gates** the implementation stories.

---

## Out of Scope (V1)

- **Vocabulary harmonization:** Matching company-specific terminology in generated content (e.g., "Customer Segment" vs. "Target User"). Explicitly deferred to v2.
- **Team-level template governance:** Admin controls, enforced templates, or org-level distribution. V1 is individual PM, project-level only.
- **Multiple templates:** Only one `prd-template.md` per project root. Per-feature or per-audience template selection is out of scope.
- **Template validation:** No linting, schema checking, or format enforcement. Bring-your-own markdown file.
- **Other artifact types:** This feature applies to PRD generation only. Stories, GTM, and analytics output are unaffected by the template file.

---

## Open Questions

1. **Silent detection or announced?** FR-001 specifies a notification when the template is found. Should there also be a way for the PM to bypass the template for a single run (e.g., `--no-template` flag or a prompt at detection time)?

2. **Challenger and template tension:** If the challenger review raises a flag on a section that exists in the template (e.g., the challenger questions whether a "Legal Review" section is in scope), how should this interaction be handled? Should the challenger be aware that certain sections are template-mandated?

3. **Default template scaffolding:** Should the tool offer a `/pm:template init` command that writes a default `prd-template.md` to the project root? This would help PMs who don't have a template yet but want the structure. Deferred but worth tracking.

4. **Prototype spike scope:** How many template structures are needed to feel confident about the agent's behavior? Recommendation is 2–3, but teams may have significantly different templates.
